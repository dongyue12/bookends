### 默认分支不是 main？

旧版 Git 默认创建 `master` 分支，而 GitHub/GitLab 主流已改用 `main`。两种解决方法：

```bash
# 方法一：初始化后直接改名（一次性）
git init
git branch -M main

# 方法二：设置全局默认分支名（一劳永逸）
git config --global init.defaultBranch main
```

之后创建新仓库默认就是 `main` 了。

## 分支策略

大厂常用的分支模型主要有两种：**Trunk-Based Development**（主流）和 **Git Flow**（传统）。

### Trunk-Based Development（推荐）

Google、Facebook、多数互联网公司的主流选择。核心思想：**所有人都往 main 提交，分支存活不超过 1 天**。

```
main ───●────●────●────●────●────●──▶
         │    │         │
         │    │         └── 新功能 C（分支存活 4 小时）
         │    └── 新功能 B（分支存活 8 小时）
         └── 新功能 A（分支存活 6 小时）
```

**特点：**
- 分支短命（≤ 1 天），减少合并冲突
- 未完成功能用 feature flag 隐藏，不影响线上
- 高频集成，CI/CD 自动化测试和部署
- Code Review 通过 PR/MR 完成

**适用场景：** CI/CD 成熟、测试覆盖率高、迭代快的团队。

### Git Flow（传统方案）

适合有固定发布周期的项目（如客户端软件、SDK）。

```
main     ──●──────────────●──────────────●──▶
            │              │              │
develop  ──●──●──●──●──●──●──●──●──●──●──●──▶
            │  │      │
feature/A ──┘  │      │
               │      │
release/1.0 ───┘  ●──●──┘
                      │
hotfix/xxx ───────────●──┘
```

**分支类型：**

| 分支 | 来源 | 合入 | 说明 |
|------|------|------|------|
| `main` | — | — | 生产环境代码，只接受 release 和 hotfix 合入 |
| `develop` | — | — | 开发主线，所有 feature 合入这里 |
| `feature/*` | develop | develop | 新功能开发 |
| `release/*` | develop | main + develop | 发布前冻结，只修 bug |
| `hotfix/*` | main | main + develop | 线上紧急修复 |

**适用场景：** 有固定版本号、长发布周期（如每两周发版）的项目。

## 分支操作

```bash
git branch                              # 查看本地分支列表
git branch -r                           # 查看远程分支列表
git branch -a                           # 查看所有分支
git branch <name>                       # 创建分支（不切换）
git checkout <name>                     # 切换分支
git checkout -b <name>                  # 创建并切换
git switch <name>                       # 切换分支（Git 2.23+，语义更清晰）
git switch -c <name>                    # 创建并切换（推荐写法）
git branch -m <old> <new>               # 重命名分支
git branch -d <name>                    # 删除已合并分支
git branch -D <name>                    # 强制删除分支
git push origin --delete <name>         # 删除远程分支
```

### 合并策略

```bash
# merge：保留完整分支历史（默认）
git merge feature/login

# squash merge：把整个分支的提交压缩成一个（提交历史干净）
git merge --squash feature/login
git commit -m "feat: 添加登录功能"

# rebase：把当前分支的提交"搬"到目标分支顶部（历史线性）
git rebase main
```

| 方式 | 历史形态 | 适用场景 |
|------|---------|---------|
| merge | 保留分支拓扑 | 多人协作的功能分支 |
| squash merge | 一条提交 | PR 合入 main（推荐） |
| rebase | 线性历史 | 同步上游变更 |

> **大厂惯例：** PR 合入 main 时用 **squash merge**，保持 main 历史是一条干净的直线。日常同步用 **rebase**。

## 撤销与回滚

撤销操作分为四个层级：**工作区 → 暂存区 → 本地提交 → 远程提交**，逐级递进。

### 层级一：撤销工作区修改（文件还没 add）

```bash
git restore <file>                   # 丢弃单个文件的修改
git restore .                        # 丢弃所有文件的修改
git checkout -- <file>               # 旧版写法，效果相同
```

> 本质是从暂存区（或最新提交）恢复文件。**不可逆**，恢复后修改永久丢失。

### 层级二：撤销暂存区（已经 add 了，还没 commit）

```bash
git restore --staged <file>          # 取消单个文件的暂存，修改回到工作区
git restore --staged .               # 取消所有暂存
git reset HEAD <file>                # 旧版写法，效果相同
```

> 只是把文件从暂存区拿出来，**工作区的修改还在**。

### 层级三：撤销本地提交（已经 commit 了，还没 push）

#### reset — 回退到指定提交

```
工作区 ← 暂存区 ← 本地仓库 ← 远程
                     ↑
                   这里
```

三种模式：

```
git reset --soft <commit>     修改保留在暂存区（绿色，可以直接 commit）
git reset --mixed <commit>    修改保留在工作区（红色，需要重新 add）
git reset --hard <commit>     修改全部丢弃（不可恢复）
```

**`<commit>` 的写法：**

| 写法 | 含义 |
|------|------|
| `HEAD~1` | 回退 1 次提交 |
| `HEAD~3` | 回退 3 次提交 |
| `<hash>` | 回退到指定的那个提交 |
| `origin/main` | 回退到远程 main 的最新状态 |

#### 具体示例

当前提交历史：
```
A --- B --- C --- D (HEAD)
```

回退到 B（即撤销 C 和 D）：

```bash
# 方式一：用相对引用
git reset --soft HEAD~2     # 撤销 C、D，修改回到暂存区
git reset --mixed HEAD~2    # 撤销 C、D，修改回到工作区
git reset --hard HEAD~2     # 撤销 C、D，修改彻底丢弃

# 方式二：用 commit hash（回退到指定提交 B）
git log --oneline           # 先查到 B 的 hash
git reset --hard abc1234    # 直接跳到 abc1234

# 方式三：回退到远程版本（放弃本地所有提交）
git fetch origin
git reset --hard origin/main
```

#### reset 三种模式对比

```
假设：工作区有文件 f，暂存区有文件 f（v2 版本），最新提交是 f（v2）

git reset --soft HEAD~1   →  f 在暂存区（v2），工作区不变（v2）
                             适合：把多个 commit 合并成一个

git reset --mixed HEAD~1  →  f 不在暂存区，在工作区（v2）
                             适合：重新整理后再 commit（默认）

git reset --hard HEAD~1   →  f 全部丢弃，回到 HEAD~1 的状态
                             适合：彻底放弃最近的修改
```

#### commit --amend（修改最后一次提交）

```bash
git commit --amend -m "新的提交信息"       # 改信息
git commit --amend --no-edit             # 追加修改，不改信息
```

> 本质是把新修改"塞进"上一次提交，不产生新 commit。只适用于**还没 push** 的提交。

### 层级四：撤销远程提交（已经 push 了）

已经 push 的提交**不能用 reset**（会破坏别人的历史），要用 `revert`。

```bash
# 撤销某次提交，创建一个新的"反向"提交
git revert abc1234

# 撤销多个连续提交
git revert abc1234..def5678 --no-commit  # 先不提交，确认后手动 commit

# 撤销最近 3 次提交（一次 revert 一个）
git revert HEAD~2..HEAD
```

> `revert` 不删除原提交，而是创建一个新的"反向操作"提交。历史完整、安全、可追溯。

### 清除工作区

#### 清除未跟踪文件

```bash
git clean -n                    # 预览：列出会被删除的文件（安全，不实际删除）
git clean -f                    # 删除未跟踪的文件
git clean -fd                   # 删除未跟踪的文件和目录
git clean -fx                   # 删除未跟踪的文件，包括 .gitignore 里的
```

#### 清除所有未提交的修改（回到干净状态）

```bash
# 一步到位：丢弃工作区 + 暂存区所有修改
git reset --hard HEAD

# 连未跟踪文件一起清
git reset --hard HEAD && git clean -fd
```

## 切换到指定版本（只读，不回退）

和 reset/revert 不同，切换只是**让工作区暂时变成某个历史版本的样子**，不移动分支指针，不改变任何提交历史。

```bash
# 切换到指定提交（进入 "detached HEAD" 状态）
git checkout <commit-hash>

# 查看代码、运行测试、验证某个版本的行为...
# 验证完后切回正常分支
git checkout main                   # 或 git switch main
```

### detached HEAD 是什么？

```
正常状态：                          detached HEAD：
HEAD → main → D --- C --- B --- A       HEAD → B --- A
                                                  ↑
                                          main → D --- C
```

HEAD 直接指向某个 commit 而不是分支。你在这个状态下做的任何提交，切回分支后就"找不到"了（除非记下 hash）。

### 想基于历史版本做修改怎么办？

```bash
# 从历史版本创建一个新分支
git checkout -b fix-from-old <commit-hash>

# 或两步：先切过去，再建分支
git checkout <commit-hash>
git switch -c my-branch
```

### 切换 vs 回退 对比

| | `git checkout <hash>` | `git reset --hard <hash>` |
|---|---|---|
| 分支指针 | 不动 | **移动**到指定提交 |
| 工作区内容 | 变成指定版本的样子 | 变成指定版本的样子 |
| 后续提交 | 需要先建分支，否则提交会丢失 | 可以直接提交，从该点继续 |
| 已有提交 | 保留，一个不少 | 指定提交**之后**的全部丢弃 |
| 能 push 吗 | 不能（不在分支上） | 能，但需要 `--force`（危险） |
| 适用场景 | 临时看看历史代码、排查 bug | 确认要永久丢弃后面的提交 |

## 临时保存（stash）

```bash
git stash                                # 保存当前工作进度
git stash save "描述"                    # 带描述保存
git stash list                           # 查看 stash 列表
git stash pop                            # 恢复最近一次 stash 并删除
git stash apply                          # 恢复但不删除
git stash drop                           # 删除最近一次 stash
git stash pop stash@{1}                  # 恢复指定的 stash
```

## 常用组合

```bash
# 看最近一周谁改了什么
git log --oneline --since="1 week ago" --author="你的名字"

# 撤销最近一次 commit（保留修改）
git reset --soft HEAD~1

# 把当前分支的提交整理成一条
git rebase -i HEAD~3       # 交互式整理最近 3 次提交

# 对比两个分支的差异
git diff main..feature/xxx

# 查看哪些文件在 main 上改了但我还没同步
git fetch origin
git diff HEAD..origin/main --name-only
```
